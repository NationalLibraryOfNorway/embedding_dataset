# Synthetic Embedding Dataset

A synthetic dataset generation pipeline for text embedding tasks, implementing the two-phase LLM prompting approach from [Wang et al. (2024)](https://arxiv.org/abs/2401.00368).

## Overview

This pipeline generates 50,000 training examples per task type across five data categories — retrieval, classification, short text matching, long text matching, and semantic textual similarity — in Danish, Swedish, and Norwegian. For each task, data is generated by [Google Gemma-4-31B-IT](https://huggingface.co/google/gemma-4-31B-it) running locally via [vLLM](https://github.com/vllm-project/vllm). Each individual task dataset is pushed to the [NbAiLab Hugging Face](https://huggingface.co/NbAiLab). A dataset containing all the synthetic samples can be found on [DDSC's Hugging Face](https://huggingface.co/datasets/DDSC/nordic-embedding-training-data). 

## Background

Wang et al. (2024) introduce a method for training high-quality text embedding models using **only LLM-generated synthetic data**, eliminating the need for complex multi-stage pre-training pipelines. The core insight is that a capable LLM can generate diverse, high-quality (query, positive document, hard negative) triples when guided by carefully designed prompts.

The generation follows a **two-step prompting strategy**:

1. **Brainstorm** — prompt the LLM to generate ~20 task definitions for a given category (e.g., "Retrieve documents that answer FAQ-style queries on children's nutrition").
2. **Generate** — for each task definition, prompt the LLM to produce one concrete data example with randomised placeholder variables (document length, difficulty, clarity, etc.) to maximise diversity.

This repo reproduces that approach using Gemma-4-31B-IT instead of GPT-4, extended to Swedish and Norwegian.

> Wang, L., Yang, N., Huang, X., Yang, L., Majumder, R., & Wei, F. (2024). Improving Text Embeddings with Large Language Models. *arXiv:2401.00368*.

## Generated Data Types

### 1. Retrieval — Short-Long (Table 8)

A short user query paired with a long relevant document and a hard negative document that appears relevant but is not.

**Schema:**
```json
{
  "user_query": "short search query",
  "positive_document": "relevant document (≥50–500 words)",
  "hard_negative_document": "misleading but partially relevant document"
}
```

**Placeholder variables (randomly sampled per example):**

| Variable | Values |
|---|---|
| `{num_words}` | `50`, `100`, `200`, `300`, `400`, `500` |
| `{query_type}` | `extremely long-tail`, `long-tail`, `common` |
| `{query_length}` | `less than 5 words`, `5 to 15 words`, `at least 10 words` |
| `{difficulty}` | `high school`, `college`, `PhD` |
| `{clarity}` | `clear`, `understandable with some effort`, `ambiguous` |

**Example** (from Wang et al., Table 16):
```
user_query: "How to use Microsoft Power BI for data analysis"

positive_document: "Microsoft Power BI is a sophisticated tool that requires time
and practice to master. In this tutorial, we'll show you how to navigate Power BI
and perform basic data analysis. First, launch the application and log in with a
Microsoft account. Next, create a new project using the Wizard on the left-hand
toolbar. Import the dataset you wish to work with by clicking on 'Get Data'. Power
BI can handle a variety of file formats, including Excel and CSV. Load your data,
then use a sequence of drag-and-drop operations to generate charts and maps."

hard_negative_document: "Excel is an incredibly powerful tool for managing and
analyzing large amounts of data. This tutorial walks you through transferring data
to a spreadsheet, colour-coding cells, and using VLOOKUP, conditional formatting,
and pivot tables..."
```

### 2. Classification — Long-Short (Table 9)

A longer input text paired with its correct classification label and a misleading label that is related but incorrect.

**Schema:**
```json
{
  "input_text": "text to classify (length controlled by num_words)",
  "label": "correct label",
  "misleading_label": "incorrect but related label"
}
```

**Placeholder variables:**

| Variable | Values |
|---|---|
| `{num_words}` | `less than 10`, `at least 10`, `at least 50`, `at least 100`, `at least 200` |
| `{difficulty}` | `high school`, `college`, `PhD` |
| `{clarity}` | `clear`, `understandable with some effort`, `ambiguous` |

**Example** (from Wang et al., Table 16):
```
input_text: "I purchased a TV from your store about a month ago, and it has
completely stopped working. The screen stays black even when I turn it on..."

label: "High Severity"
misleading_label: "Low Severity"
```

---

### 3. Short Text Matching — Short-Short (Table 10)

Both the input and the matching document are very short (a sentence or phrase). No hard negative is generated — the task is already sufficiently difficult without one.

**Schema:**
```json
{
  "input": "short phrase or sentence",
  "positive_document": "semantically matching short phrase or sentence"
}
```

**Example** (from Wang et al., Table 16):
```
input:             "I'm going to make him an offer he can't refuse."
positive_document: "The Godfather"
```

Other representative task types for this category include:
- Match a scientific paper title to a citing paper's title
- Match a word to its definition
- Match a notable person's name to their occupation or achievement

---

### 4. Long Text Matching — Long-Long (Table 11)

Both the input and the matching document are long (≥300 words each) with minimal word overlap. No hard negative is generated (omitted for API latency reasons).

**Schema:**
```json
{
  "input": "long document (≥300 words)",
  "positive_document": "semantically related long document (≥300 words)"
}
```

**Example** (from Wang et al., Table 16):
```
input: "Before the Superior Court of California, County of Los Angeles. In the
matter of the lawsuit initiated by the Plaintiff Jane Smith against the Defendants
PQR Industries and LMN Enterprises, charging they negligently designed, produced,
and advertised a product that directly caused harm to her..."

positive_document: [A legally similar brief with comparable negligence arguments,
different parties and facts but structurally and thematically parallel]
```

Representative task types for this category include:
- Given a document supporting a debatable argument, find one containing opposing arguments
- Given a lengthy business proposal, retrieve competitive business strategies in the same industry

---

### 5. Unit Triple — Monolingual STS (Table 12)

Three sentences/phrases/passages where S1–S2 have high semantic similarity and S1–S3 have lower semantic similarity, on a 1–5 scale. No task definition brainstorming step is needed — the prompt is self-contained.

**Schema:**
```json
{
  "S1": "base unit",
  "S2": "unit with high similarity to S1",
  "S3": "unit with low similarity to S1"
}
```

**Placeholder variables:**

| Variable | Values |
|---|---|
| `{unit}` | `sentence`, `phrase`, `passage` |
| `{high_score}` | `4`, `4.5`, `5` |
| `{low_score}` | `2.5`, `3`, `3.5` |
| `{difficulty}` | `elementary school`, `high school`, `college` |

**Example** (from Wang et al., Table 16):
```
S1: "Tom loves to ride his blue bicycle in the park every morning."
S2: "Every morning, Tom enjoys riding his blue bike in the park."   (similarity ≈ 4.5)
S3: "Tom takes his blue pen to school every day."                   (similarity ≈ 3.0)
```

## Prompts

### Phase 1: Task Brainstorming

These prompts generate a pool of task definitions that seed Phase 2.

#### Retrieval tasks (Table 8)
```
Brainstorm a list of potentially useful text retrieval tasks.
Here are a few examples for your reference:
- Retrieve relevant documents for a short keyword web search query that asks for
  weather information.
- Search for documents that answers a FAQ-style query on children's nutrition.
Please adhere to the following guidelines:
- Specify what the query is, and what the desired documents are.
- Each retrieval task should cover a wide range of queries, and should not be too specific.
Your output must always be a python list of strings only, with about 20 elements, and each
element corresponds to a distinct retrieval task in one sentence. Do not explain yourself
or output anything else. Be creative!
```

#### Classification tasks (Table 9)
```
Brainstorm a list of potentially useful text classification tasks.
Please adhere to the following guidelines:
- Tasks should cover a diverse range of domains and task types.
Your output must always be a Python list of strings only, with about 20 elements, and each
element corresponds to a distinct text classification task in one sentence. Do not explain
yourself or output anything else. Be creative!
```

#### Long text matching tasks (Table 10 / 11)
```
Brainstorm a list of text matching tasks where the queries are long documents.
Here are a few examples:
- Given a document that supports a debatable argument, find another document that contains
  opposite arguments.
- Provided a lengthy business proposal, retrieve competitive business strategies in the
  same industry.
Your output must always be a python list of strings only, with about 20 elements, and each
element corresponds to a distinct task in one sentence. Do not explain yourself or output
anything else. Be creative!
```

#### Short text matching tasks (Table 10)
```
Brainstorm a list of text matching tasks where both the queries and the groundtruth
documents are very short (one or two sentences, even a short phrase).
Here are a few examples:
- Given a scientific paper title, retrieve the title of papers that cite the given paper.
- Match a word with its definition.
- Provided a notable person's name, identify their occupation or achievement.
Your output must always be a python list of strings only, with about 20 elements, and each
element corresponds to a distinct task in one sentence. Do not explain yourself or output
anything else. Be creative!
```

### Phase 2: Data Generation

These prompts use a task definition from Phase 1 (or are self-contained for STS) and randomly sampled placeholder values to produce one example per call.

#### Retrieval (Table 8)
```
You have been assigned a retrieval task: {task}
Your mission is to write one text retrieval example for this task in JSON format. The JSON
object must contain the following keys:
- "user_query": a string, a random user search query specified by the retrieval task.
- "positive_document": a string, a relevant document for the user query.
- "hard_negative_document": a string, a hard negative document that only appears relevant
  to the query.
Please adhere to the following guidelines:
- The "user_query" should be {query_type}, {query_length}, {clarity}, and diverse in topic.
- All documents must be created independent of the query. Avoid copying the query verbatim.
  It's acceptable if some parts of the "positive_document" are not topically related to
  the query.
- All documents should be at least {num_words} words long.
- The "hard_negative_document" contains some useful information, but it should be less
  useful or comprehensive compared to the "positive_document".
- Both the query and documents should be in {language}.
- Do not provide any explanation in any document on why it is relevant or not relevant to
  the query.
- Both the query and documents require {difficulty} level education to understand.
Your output must always be a JSON object only, do not explain yourself or output anything
else. Be creative!
```

#### Classification (Table 9)
```
You have been assigned a text classification task: {task}

Your mission is to write one text classification example for this task in JSON format. The
JSON object must contain the following keys:
- "input_text": a string, the input text specified by the classification task.
- "label": a string, the correct label of the input text.
- "misleading_label": a string, an incorrect label that is related to the task.

Please adhere to the following guidelines:
- The "input_text" should be {num_words} words and diverse in expression.
- The "misleading_label" must be a valid label for the given task, but not as appropriate
  as the "label" for the "input_text".
- The values for all fields should be in {language}.
- Avoid including the values of the "label" and "misleading_label" fields in the
  "input_text", that would make the task too easy.
- The "input_text" is {clarity} and requires {difficulty} level education to comprehend.

Your output must always be a JSON object only, do not explain yourself or output anything
else. Be creative!
```

#### Long text matching (Table 11)
```
You have been assigned a text matching task: {task}
Your mission is to write one example for this task in JSON format. The JSON object must
contain the following keys:
- "input": a string, a random input specified by the task.
- "positive_document": a string, a relevant document for the "input" according to the task.
Please adhere to the following guidelines:
- The values of all fields should be in {language}.
- Both the "input" and "positive_document" should be long documents (at least 300 words),
  avoid substantial word overlaps, otherwise the task would be too easy.
- The "input" and "positive_document" should be independent of each other.
Your output must always be a JSON object only, do not explain yourself or output anything
else. Be creative!
```

#### Short text matching (Table 10)
```
You have been assigned a text matching task: {task}
Your mission is to write one example for this task in JSON format. The JSON object must
contain the following keys:
- "input": a string, a random input specified by the task.
- "positive_document": a string, a relevant document for the "input" according to the task.
Please adhere to the following guidelines:
- The values of all fields should be in {language}.
- Both the "input" and "positive_document" should be very short (a sentence or a phrase),
  avoid substantial word overlaps, otherwise the task would be too easy.
- The "input" and "positive_document" should be independent of each other.
Your output must always be a JSON object only, do not explain yourself or output anything
else. Be creative!
```

#### Unit triple / STS (Table 12)
```
Write a {unit} triple with varying semantic similarity scores in JSON format. The semantic
similarity score ranges from 1 to 5, with 1 denotes least similar and 5 denotes most similar.
Please adhere to the following guidelines:
- The keys in JSON are "S1", "S2", and "S3", the values are all strings in {language}, do
  not add any other keys.
- There should be some word overlaps between all three {unit}s.
- The similarity score between S1 and S2 should be {high_score}.
- The similarity score between S1 and S3 should be {low_score}.
- The {unit}s require {difficulty} level education to understand and should be diverse in
  terms of topic and length.
Your output must always be a JSON object only with three keys "S1", "S2" and "S3", do not
explain yourself or output anything else. Be creative!
```

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1 — Task Generation  (generate_tasks.py)             │
│                                                             │
│  generate_retrieval_tasks.py           ─┐                   │
│  generate_text_classification_tasks.py ─┤  ~20 tasks/call   │
│  generate_text_matching_tasks_long.py  ─┤  × N calls        │
│  generate_text_matching_tasks_short.py ─┘                   │
│                    │                                        │
│             post_processing.py  (clean + deduplicate)       │
│                    │                                        │
│           HuggingFace Hub  (NbAiLab/*-processed)            │
└─────────────────────────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 2 — Data Generation  (generate_data.py <language>)   │
│                                                             │
│  generate_from_retrieval_task.py                            │
│  generate_from_classification_task.py   each generates      │
│  generate_from_text_matching_long.py    50,000 samples      │
│  generate_from_text_matching_short.py   per language        │
│  generate_from_unit_triple.py                               │
│                    │                                        │
│       CSV files  +  HuggingFace Hub  (NbAiLab/*)            │
└─────────────────────────────────────────────────────────────┘
```

Each Phase 2 generator (defined in `generators.py`) calls the Gemma model in batches via vLLM, randomly sampling placeholder values to diversify the outputs, then converts results to a Hugging Face `Dataset` and optionally pushes to the Hub.

## Usage

### Prerequisites

```bash
uv sync
```

A running OpenAI-compatible API server is required. For local inference, start a vLLM server:

```bash
vllm serve google/gemma-4-31B-it --port 8000
```

The generation scripts connect to the server via environment variables (defaults shown):

```bash
export VLLM_BASE_URL="http://localhost:8000/v1"  # or any OpenAI-compatible endpoint
export VLLM_API_KEY="dummy"                       # real key needed for hosted APIs
```

To use a hosted API (e.g., OpenAI, Together, Fireworks), just set the appropriate URL and key — no code changes needed.

### Full pipeline (Phase 1 + Phase 2, Norwegian Bokmål)

```bash
python src/main.py
```

### Phase 1 only — generate task definitions

```bash
python src/generate_tasks.py
```

This brainstorms task definitions for all four task categories and pushes cleaned task lists to the Hugging Face Hub.

### Phase 2 only — generate data for a specific language

```bash
python src/generate_data.py "NORWEGIAN BOKMÅL"
python src/generate_data.py "NORWEGIAN NYNORSK"
python src/generate_data.py SWEDISH
python src/generate_data.py DANISH
```

### Generate all languages

```bash
python src/generate_all_languages.py
```

## Configuration

All key parameters live in `variables.py`:

| Variable | Default | Description |
|---|---|---|
| `model_id` | `google/gemma-4-31B-it` | Model name to request from the API |
| `base_url` | `http://localhost:8000/v1` | OpenAI-compatible API endpoint (env: `VLLM_BASE_URL`) |
| `api_key` | `dummy` | API key (env: `VLLM_API_KEY`) |
| `hf_user` | `NbAiLab` | Hugging Face user for dataset pushes |
| `total_desired_samples` | `50000` | Samples to generate per data type per language |
| `temperature` | `1.0` | LLM sampling temperature |
| `top_p` | `1.0` | LLM top-p sampling |
| `push_to_hf` | `True` | Whether to push generated datasets to the Hub |

## Output Datasets

Generated datasets are pushed to the Hugging Face Hub under `NbAiLab/`:

| Type | Dataset ID pattern |
|---|---|
| Task definitions (all types) | `NbAiLab/{type}-tasks-processed` |
| Retrieval | `NbAiLab/synthetic-from-retrieval-tasks-{language}` |
| Classification | `NbAiLab/synthetic-from-classification-tasks-{language}` |
| Long matching | `NbAiLab/synthetic-from-text-matching-long-tasks-{language}` |
| Short matching | `NbAiLab/synthetic-from-text-matching-short-tasks-{language}` |
| Unit triple / STS | `NbAiLab/synthetic-from-unit-triple-tasks-{language}` |

Each row in the generated datasets contains:

| Column | Content |
|---|---|
| `response` | JSON string with the generated example |
| `model` | Model ID (`google/gemma-4-31B-it`) |
| `prompt` | The full prompt sent to the model |
